## Recommended execution order

1. **001** first — it introduces the `--ease-out`/`--ease-in-out` tokens in `css/main.css`'s `:root`. All other plans that touch `css/main.css` should land on top of it.
2. **002** next — also touches `css/main.css` and references `--ease-out`. Written to be independent of 001 (it adds the token itself if missing), but running 001 first avoids a trivial merge overlap in the same `:root` block.
3. **003** — fully independent, touches only `js/theme.js`. Can run in any order relative to the others.
4. **004** — fully independent, touches only `newsletter/007/index.html`. Can run in any order relative to the others.

## Dependencies

- **001 and 002 both edit `css/main.css`'s `:root` block.** If both are executed by separate agents/sessions without 001 landing first, whoever runs second should check for an existing `--ease-out` definition before adding a duplicate (both plans include this check in their steps).
- **003 and 004 have no file overlap with any other plan** — safe to run fully in parallel with 001/002.

## Notes

- Plan 004 was written only after confirming via `git log` that the Estadio interstitial's removal (commit `af3bc7f`, "007: dateline masthead intro + real→ghibli scroll dissolve") was a deliberate redesign, not a regression — the plan deletes leftover dead code, it does not undo that redesign decision.
- `projects/merfish/index.html` already defines its own `--ease-out` token and is intentionally excluded from Plan 001's scope.
